## 2. Порядок разработки (приоритеты)

### Фаза 0: Завершить фундамент (1 неделя)

**Приоритет: КРИТИЧЕСКИЙ**

1. **Довести StateMachine до рабочего состояния**
   - Сейчас: скелеты без логики, StateMachine создается но не запускается
   - Нужно: реализовать переходы между состояниями, запуск начального состояния в AppBootstrap
   - Минимум: Loading -> Initializing -> Menu (три работающих перехода)

2. **Реализовать сервис-локатор или простой DI**
   - Не использовать синглтоны напрямую — потом сложно тестировать и рефакторить
   - Варианты: простой ServiceLocator, VContainer (бесплатный DI для Unity), или ручной DI через конструкторы
   - Рекомендация: начать с простого ServiceLocator, мигрировать на VContainer позже если нужно

3. **Создать базовую систему событий**
   - C# events или простая event bus
   - Нужна для связи между модулями без прямых зависимостей

4. **Настроить сцены**
   - Переименовать SampleScene -> Bootstrap (или создать отдельную)
   - Решить: одна сцена с переключением экранов через UI или несколько сцен
   - Рекомендация для казуалки: одна сцена + UI экраны (проще, быстрее загрузка)

### Фаза 1: Прототип (2-3 недели)

**Приоритет: ВЫСОКИЙ**

Порядок внутри фазы:
1. Data Structures (MoleculeData, LevelData, GraphData) — 2 дня
2. Domain Logic (Molecule, MoleculeGraph, GameRules) — 3 дня

## 3. Технические рекомендации

### 3.1 Архитектура Core Logic

**Отделить игровую логику от Unity:**
- Классы `Molecule`, `MoleculeGraph`, `GameRules`, `TurnExecutor` не должны наследовать MonoBehaviour
- Это позволит писать Unit тесты без Unity Test Runner
- View-компоненты (MoleculeView) будут MonoBehaviour и подписываться на изменения моделей

**Пример разделения:**
```
Molecule (POCO)        <->  MoleculeView (MonoBehaviour)
MoleculeGraph (POCO)   <->  GraphView (MonoBehaviour)
GameRules (static)     <->  GameManager (MonoBehaviour)
```

### 3.2 Граф молекул

**Структура данных:**
- Adjacency List (словарь: ID молекулы -> список соседей)
- Не Matrix — для разреженных графов (5-20 узлов) это избыточно

**Layout:**
- Для линейных уровней: позиции задавать вручную в LevelData (ScriptableObject)
- Для бесконечного режима: Force-Directed Layout (позже)
- Рекомендация: начать с ручных позиций, автоматический layout — только для бесконечного режима

### 3.3 Система типов молекул

**Использовать Strategy Pattern:**
- `IDecrement` — стратегия декремента (простая: -1, паразит: -N)
- `IAbility` — on-click абилка (null для простых, HealNeighbors для якоря)
- `IPassiveProperty` — пассивка (null для простых, ShieldProperty для щита)

**Не делать наследование молекул (Molecule -> ParasiteMolecule -> ShieldMolecule)**
- Это приведет к проблемам с комбинациями
- Вместо этого: одна Molecule с набором стратегий, определяемых MoleculeType (enum + Factory)

### 3.4 Система видимости

**Реализация:**
- `Molecule.IsRevealed` — boolean, по умолчанию false
- Правила раскрытия: при удалении соседа, при использовании сканера, или если задано в LevelData
- MoleculeView читает IsRevealed и показывает/скрывает значение

### 3.5 Save System

**Для MVP:**
- JSON сериализация в `Application.persistentDataPath`
- Не PlayerPrefs (ограничение размера, нет структуры)
- Структура SaveData: прогресс раскрасок (пройденные уровни, открытые фрагменты), настройки, рекорд бесконечного режима

**Шифрование — НЕ нужно для MVP:**
- Это казуальная головоломка, читерство не критично
- Добавить позже если станет актуально

## 4. Рекомендации по процессу

### 4.1 Итеративная разработка

- Делать рабочую сборку после каждого спринта (1-2 недели)
- Тестировать на реальном Android устройстве, не только в редакторе
- Каждый чекпоинт должен быть играбельным (пусть с placeholder UI)

### 4.2 Тестирование

**Unit тесты (приоритетно):**
- GameRules (CheckWin, CheckLose)
- TurnExecutor (порядок выполнения шагов)
- MoleculeGraph (AddMolecule, RemoveMolecule, GetNeighbors)
- Каждый тип декремента и абилки

**Play Mode тесты (опционально):**
- Прохождение уровня от начала до конца
- Сканер (активация, показ значений)

### 4.3 Контроль версий

- Коммитить часто, маленькими порциями
- Одна фича = одна ветка (feature/state-machine, feature/molecule-core)
- Не коммитить .meta файлы без соответствующих ассетов

### 4.4 Работа с ассетами

**Шрифты:**
- Выбрать один бесплатный шрифт до начала работы с UI (Google Fonts: Orbitron, Exo 2, Rajdhani)
- Проверить лицензию (OFL — Open Font License)

**Звуки:**
- Не блокировать разработку отсутствием звуков
- Использовать placeholder звуки из бесплатных библиотек (freesound.org)
- Заменить на финальные в Фазе 3

**Шейдеры:**
- Shader Graph для всех кастомных шейдеров (Glow, ConnectionWave)
- Не писать HLSL руками — Shader Graph проще поддерживать и оптимизировать

## 5. Потенциальные риски и как их избежать

### 5.1 Scope Creep (расползание объема)
- **Риск:** Хочется добавить все типы молекул и оба режима сразу
- **Решение:** Строго следовать MVP — простые молекулы + паразит + 15 уровней = первый играбельный билд
- Щит, якорь, бесконечный режим — только после рабочего MVP
- Раскраска — это основная кампания (не отдельный режим), но начальный прототип можно тестировать без неё

### 5.2 Дизайн уровней
- **Риск:** Создание 30 сбалансированных уровней — это долго и требует плейтестов
- **Решение:** Начать с 5 уровней, дать поиграть 2-3 людям, итерировать
- Можно создать простой level editor в Unity Inspector (ScriptableObject + custom editor)

### 5.3 Производительность графа
- **Риск:** При 15-20 молекулах + анимации + частицы могут быть просадки на слабых устройствах
- **Решение:** Object Pooling с первого дня, профилировать на реальном устройстве со спринта 1.2

### 5.4 Монетизация
- **Риск:** Интеграция AdMob/IAP занимает больше времени чем кажется (ревью Google, настройка аккаунта)
- **Решение:** Создать аккаунт разработчика Google Play и AdMob аккаунт в первую неделю (пока идет Фаза 0)
